## 实验 5.2：最小 Jerk 足端轨迹 + 简化 IK

### 1. 实验目的

在实验 5.1 的基础上，进一步把参考从“关节空间升余弦包络”升级为“足端空间最小 jerk 轨迹”，并通过简化 IK 映射到关节角，验证这种方式是否能够进一步降低前进过程中的抖动、改善摆脚平滑性，并提升回放观感。

### 2. 改动思路

与 5.1 相比，这次不再直接使用 `final_swing_joint_delta_pos` 生成关节参考，而是：

1. 基于 `phase` 区分左右腿摆动/支撑相位；
2. 对摆动腿生成最小 jerk 足端轨迹；
3. 再通过简化 IK 将足端轨迹映射为 12 维关节参考；
4. 训练仍沿用原有奖励结构，仅通过参考轨迹改变学习目标。

### 3. 本次改动文件

- `humanoid/envs/x1/x1_dh_stand_env.py`
  - 新增 `_min_jerk()`
  - 新增 `_solve_2d_leg_ik()`
  - 重写 `compute_ref_state()`
- `humanoid/envs/x1/x1_dh_stand_config.py`
  - 增加 `default_foot_pos`
  - 增加 `swing_height`
  - 增加 `step_length`
  - 增加 IK 相关参数

| IK 映射 | ✅ 新增简化 IK |
| 奖励函数 | ❌ 暂未改动 |

### 6. 风险与注意事项

- 当前 IK 为简化版本，不是严格的 X1 解析几何 IK；
- `default_foot_pos` 必须与机器人零位姿态一致，否则参考会偏；
- 若 `step_length` 或 `swing_height` 设得过大，可能导致摆脚不自然或训练不稳定；
- 实验二数据显示膝关节弯曲偏小，说明当前轨迹对膝屈曲的约束不够，需要在 IK / 参考轨迹里显式增加膝关节抬腿幅度；
- 训练前建议先小步长验证，再逐步放大步幅。

### 7. 实验结果（正式结论）

本次实验使用实验二的诊断数据 `czy/data/isaac_diag_20260616_122502.csv` 进行回放分析。结果表明：机器人虽然能够沿 `cmd_linear_x = 0.4` 的方向持续前进，但整体步态稳定性不足，存在较明显的周期性抖动与姿态波动；同时，摆腿阶段的膝关节屈曲幅度偏小，导致脚端抬摆不充分、落脚过渡不够自然，是影响稳定性的主要因素之一。

从数据上看：

- `cmd_linear_x` 全程保持在 `0.4`，说明命令输入稳定；
- `base_lin_vel_x` 平均约为 `0.41 m/s`，末段平均约 `0.405 m/s`，速度跟踪本身并不差；
- 但 `base_euler_x / base_euler_y` 仍存在可见波动，说明上身稳定性一般；
- 左右膝关节角度范围整体偏小，未形成足够的屈膝抬脚动作；
- 足底接触力峰值较高，且双脚接触状态切换不够平滑，说明落脚阶段仍有较强冲击或过渡不连续。

综合判断，本次“最小 jerk 足端轨迹 + 简化 IK”方案**尚未达到稳定行走目标**。虽然前进速度可以维持，但由于膝关节弯曲不足、脚步抬摆不充分以及 IK 近似带来的姿态误差，机器人步态表现出抖动、碎步和过渡生硬等问题。后续应优先增加摆动膝关节屈曲幅度，并对足端轨迹与 IK 模型进行进一步修正。

### 1. 实验目的

在**不改动奖励权重与训练超参**的前提下，仅替换 `compute_ref_state()` 中的关节参考包络函数，验证更平滑的摆动参考能否改善：

- `_reward_ref_joint_pos` 跟踪质量
- 摆动起止点的关节速度突变（PD 抖动）
- 步态相位与落脚的一致性

### 2. 对照基线（改动前）

**包络**：半波整流正弦 `|sin(2π·phase)|`

```python
sin_pos = sin(2π·phase)
左腿摆动 (sin < 0):  ref_delta = -clip(sin, max=0) · Δq
右腿摆动 (sin > 0):  ref_delta =  clip(sin, min=0) · Δq
|sin| < 0.1:         ref_delta = 0   # 双支撑
ref_dof_pos = ref_delta + default_dof_pos
```

**特点**：

- 摆动峰值在周期中点，与相位掩码一致
- 摆动起止点处包络导数不连续（尖角），易引起参考速度跳变
- 上升/下降对称，不符合「快抬慢落」的真实步态

### 3. 本次改动

**文件**：`humanoid/envs/x1/x1_dh_stand_env.py` → `compute_ref_state()`

**新包络**：升余弦（Hann 窗）

```
phase_mod = phase % 1
left_tau  = clamp(phase_mod - 0.5, min=0) × 2    # [0.5, 1.0] → [0, 1]
right_tau = clamp(phase_mod, max=0.5) × 2        # [0.0, 0.5] → [0, 1]

left_env  = (1 - cos(π·left_tau)) / 2
right_env = (1 - cos(π·right_tau)) / 2

|sin(2π·phase)| < 0.1 时 left_env / right_env = 0   # 双支撑死区保留

ref_delta[:, 0:6]  = left_env  · final_swing_joint_delta_pos[0:6]
ref_delta[:, 6:12] = right_env · final_swing_joint_delta_pos[6:12]
ref_action = 2 · ref_delta
ref_dof_pos = ref_delta + default_dof_pos
```

**改动性质**：

| 项 | 是否修改 |
|----|----------|
| 参考包络形状 | ✅ 半波正弦 → 升余弦 |
| `_get_phase()` / `cycle_time` | ❌ 不变 |
| `_get_stance_mask()` 逻辑 | ❌ 不变（仍用 sin 阈值 0.1） |
| `final_swing_joint_delta_pos` | ❌ 不变 |

## 实验 5.4：修复参考轨迹缺陷（支撑相位置 + Z 轨迹 + IK 精度 + 步长缩放）

### 1. 实验目的

在实验 5.2/5.3 的基础上，修复代码审查发现的参考轨迹缺陷，验证这些修正能否进一步提升步态稳定性。

### 2. 修复内容

#### 问题 1：支撑相足端位置未随前进更新

**问题**：支撑相足端 X 坐标始终固定在 `default_foot_pos[0]`，但机器人前进时，支撑脚相对躯干应向后移动。

**修复**：
```python
# 支撑相：足端从前进位置（end）后退到起始位置（start）
left_stance_x = self._min_jerk(left_end[:, 0], left_start[:, 0], s_left)
left_foot[:, 0] = torch.where(left_swing, 
    self._min_jerk(left_start[:, 0], left_end[:, 0], s_left), 
    left_stance_x)
```

**影响**：策略现在会学习到正确的"支撑脚向后滑动"参考，减少躯干前进时的拖拽感。

#### 问题 2：Z 轨迹使用对称三角参数

**问题**：原代码使用 `2 * min(s, 1-s)` 作为 min_jerk 参数，导致 Z 轨迹在 s=1 时回到起点高度，而非落脚点高度。

**修复**：分为上升段和下降段分别计算：
```python
# 上升段 (s < 0.5)：从 start_z 到 mid_z
# 下降段 (s >= 0.5)：从 mid_z 到 start_z（落脚点高度）
left_swing_z = torch.where(s_left < 0.5,
    self._min_jerk(left_start[:, 2], left_mid_z, 2.0 * s_left),
    self._min_jerk(left_mid_z, left_start[:, 2], 2.0 * (s_left - 0.5)))
```

**影响**：摆动相 Z 轨迹现在更符合"抬脚-平移-落脚"的物理过程。

#### 问题 3：简化 IK 系数缺乏几何验证

**问题**：原 IK 使用经验线性系数 (-0.8, -1.2, 1.6, 0.6...)，与 URDF 几何参数不符，导致膝关节角度计算不准确。

**修复**：使用余弦定理计算精确几何 IK：
```python
# 余弦定理计算膝关节角度
cos_knee = (r**2 - L1**2 - L2**2) / (2 * L1 * L2)
knee_angle = torch.acos(torch.clamp(cos_knee, -1.0, 1.0))

# 髋关节角度
alpha = torch.atan2(z, x)
cos_hip_offset = (L1**2 + r**2 - L2**2) / (2 * L1 * r)
hip_pitch = alpha + torch.acos(torch.clamp(cos_hip_offset, -1.0, 1.0))

# 踝关节角度：保持足底水平
ankle_pitch = hip_pitch + knee_angle - torch.pi / 2
```

**影响**：IK 精度显著提升，膝关节角度更符合机器人几何约束。

#### 问题 4：步长与命令速度脱钩

**问题**：`step_length` 固定为 0.08，无论 `cmd_linear_x` 是 0.4 还是 1.2。

**修复**：
```python
cmd_vel = torch.abs(self.commands[:, 0])
step_length = self.step_length * torch.clamp(cmd_vel / 0.4, 0.5, 2.0)
```

**影响**：低速时步长自动缩短，避免碎步；高速时步长适当增大。

## 实验 5.5：修复 IK Bug（右腿 hip_pitch 取负问题）

### 1. 问题发现

实验 5.4 回放时发现机器人**左腿姿态异常，左腿不朝前**。右腿正常。

通过数据分析发现左腿 `hip_yaw` 严重偏离默认值：

| 关节 | 左腿实际 | 默认值 | 偏差 |
|------|---------|--------|------|
| **hip_yaw** | **-1.050** | **-0.31** | **-0.74 rad (42°)** |
| hip_pitch | 0.042 | 0.4 | -0.358 |
| ankle_pitch | -0.388 | -0.21 | -0.178 |

策略输出的左腿 `action_hip_yaw = -1.678`，导致 `pos_des_raw = action * 0.5 + (-0.31) = -1.149`。

### 2. 根因分析

`pos_des_raw` 是策略输出的目标位置，不是 IK 参考轨迹。策略**主动学到了偏转左腿 yaw 的行为**。

**为什么会学这个？**
- 5.4 的几何 IK 对右腿 `hip_pitch` 取负（Bug），导致左右腿 pitch 参考严重不对称（左 0.036 vs 右 -0.276）
- 策略在训练中为了弥补 pitch 参考不合理，被迫通过偏转 hip_yaw 来补偿
- 形成了"左腿偏转 + 右腿正常"的畸形策略

### 3. Bug 定位

IK 函数 `_solve_2d_leg_ik()` 对右腿 `hip_pitch` 取负：

```python
return torch.stack((-hip_pitch, hip_roll, hip_yaw, knee_pitch, ankle_pitch, ankle_roll), dim=1)
```

同时，X1 腿部关节布局非标准矢状面，2D 几何 IK 无法正确处理
- 在错误的基础上取负，加剧了不对称

### 4. 修复方案（实验 5.5）

**1. IK 函数改为 Jacobian 映射**：放弃几何 IK，以默认关节角度为基础，根据足端偏移调整 pitch 关节，roll/yaw 保持默认。关键改动：
- 去掉右腿 `hip_pitch` 取负
- 左右腿独立从各自默认关节角度出发

**2. 增加奖励约束 hip_yaw/roll**：添加 `_reward_hip_yaw_roll_default()`，防止策略通过偏转 yaw/roll 补偿 pitch 误差。

### 5. 修改文件

- `humanoid/envs/x1/x1_dh_stand_env.py`
  - 重写 `_solve_2d_leg_ik()` 为 Jacobian 映射方案
  - 新增 `_reward_hip_yaw_roll_default()`
- `humanoid/envs/x1/x1_dh_stand_config.py`
  - 添加奖励权重 `hip_yaw_roll_default = 0.5`

- 左腿 hip_yaw 实际值接近默认值 -0.31（偏差 < 0.1 rad）
